## 1. 核心理念

### 1.1 文档是新的"源代码"

AI 时代的范式转变：

```
传统模式：  文档(可选) → 高级语言(人工作) → 机器语言
AI 模式：   文档/PRD(人工作) → 高级语言(Agent翻译) → 机器语言
```

开发者的核心职责从"写代码"转变为"写文档 + 监督 Agent + Review 产出"。文档是人与项目之间最重要的接口——它损失细节但保留骨架，自然语言对人类更友好，Agent 作为翻译层使文档到代码的转换成本大幅降低。

### 1.2 角色定位

| 角色 | 职责 |
|------|------|
| **开发者（你）** | 定义需求、写 PRD、审查计划、Review 代码、验收功能 |
| **Agent（Claude Code / Codex）** | 制定计划、实现代码、写测试、自查 Bug |

你是技术负责人，Agent 是执行力极强但需要明确指令的工程师。你投入最多的地方应该是**PRD 和 Review**，而不是写代码。

### 1.3 信息越充分，产出越好

Agent 的产出质量与你提供的上下文成正比。一份模糊的口头描述会产出需要反复修改的代码；一份精确的 PRD 通常一次就能产出可用的实现。

## 2. 工作流选择

拿到一个新需求时，先判断走哪条路：

```
新需求到达
    │
    ├─ 体量大 且 方向明确？
    │     │
    │     ├─ 是 → 工作流一：文档主导（六阶段）
    │     │
    │     └─ 否 → 工作流二：原型主导
    │                │
    │                └─ 开发中复杂度膨胀？
    │                      │
    │                      ├─ 否 → 继续迭代，完成即可
    │                      │
    │                      └─ 是 → 让 AI 阅读代码生成 PRD → 切换到工作流一
    │
    └─ 已有行业共识的功能？（如平台跳跃、背包系统）
          │
          └─ 是 → 工作流二也可以，因为 Agent 有足够先验知识
```

**经验法则**：
- 预计超过 3 个文件改动 → 文档主导
- 1-2 个文件的小改动 → 原型主导
- 不确定 → 先原型，膨胀了再切文档

## 3. 工作流一：文档主导（六阶段）

这是本项目的主要工作流，适用于所有模块级开发。

### 阶段一：发散需求

**目标**：从模糊想法到清晰的设计方向。

**做什么**：
1. 向 Agent 描述你的模糊想法（不需要技术细节）
2. 让 Agent 调研类似游戏/系统的设计
3. Agent 生成 2-3 个纯逻辑设计方案
4. 你选择一个方向或组合出新方案

**不做什么**：
- 不讨论技术实现
- 不写代码
- 不打开 IDE

**产出物**：一段明确的需求描述，足以开始写 PRD。

**示例对话**：
```
你：我想让游戏有一个地图系统，玩家打完一个房间后能选下一个房间
Agent：StS 的地图是分层结构，每层 1-4 个节点……我建议以下 3 种方案……
你：方案 2 不错，但我想简化一下，不需要分叉路径……
```

---

### 阶段二：写 PRD

**目标**：产出一份 Agent 可直接执行的需求文档。

**这是整个流程中人工投入最多的阶段，通常占开发时间 50% 以上。**

**PRD 必须包含的三要素**：

#### ① 术语表

统一命名，消除歧义。Agent 和团队成员看到同一个词必须理解为同一个概念。

```markdown
| 术语 | 英文 | 定义 |
|------|------|------|
| 遭遇 | Encounter | 一次战斗中出现的敌人组合 |
| 意图 | Intent | 敌人下一回合将执行的行动类型 |
```

**为什么重要**：Agent 不会主动问你"遭遇"和"战斗"有什么区别，它会自己猜——猜错了你就得花时间返工。

#### ② 交互方式

描述用户（玩家）如何与系统交互。用流程图或分步描述，不涉及代码。

```markdown
1. 玩家点击地图上的战斗节点
2. 系统根据当前幕和层数选择一组敌人
3. 进入战斗画面，敌人从左到右排列
4. 战斗结束后回到地图
```

**为什么重要**：交互方式定义了系统的"外部行为"，是验收的基准。Agent 可能实现出功能正确但交互别扭的系统。

#### ③ 验收标准

可勾选的 checklist，包括正常流程和边界情况。

```markdown
- [ ] 点击可选节点后进入对应房间
- [ ] 已访问节点不可点击
- [ ] Boss 节点固定在顶层
- [ ] 击败 Boss 后不回到地图，进入幕间流程
```

**为什么重要**：没有验收标准，你无法判断 Agent 的产出是否"完成"。验收标准也可以直接转化为测试用例。

**PRD 存放规范**：
```
Assets/DesignDoc/
├── PRD_00_MASTER.md       ← 项目总体 PRD
├── PRD_01_MAP_UI.md       ← 模块 PRD
├── PRD_02_XXX.md
└── ...
```

---

### 阶段三：Plan + Review

**目标**：让 Agent 制定实现计划，再用另一个 Agent Review，循环约 2 轮。

**步骤**：

```
第 1 轮：
┌──────────────┐    plan.md    ┌──────────────┐
│  Claude Code  │ ───────────→ │    Codex /    │
│  （制定计划）   │              │   ChatGPT    │
│               │ ←─────────── │  （Review）    │
└──────────────┘   review 反馈  └──────────────┘

第 2 轮：
┌──────────────┐   修改后 plan  ┌──────────────┐
│  Claude Code  │ ───────────→ │    Codex /    │
│ （修改计划）    │              │   ChatGPT    │
│               │ ←─────────── │  （确认/微调）  │
└──────────────┘               └──────────────┘
```

**Claude Code 中的操作**：
```
你（对 Claude Code）：
  请阅读 DesignDoc/PRD_01_MAP_UI.md，制定实现计划，输出到 DesignDoc/PLAN_01_MAP_UI.md。
  计划应包含：
  1. 需要新增和修改的文件列表
  2. 每个文件的职责和关键方法
  3. 实现步骤（按依赖顺序）
  4. 可以用 TDD 方式先写的测试
```

**Review（在 Codex / ChatGPT 中）**：
```
你（对 Codex）：
  请 Review 以下实现计划，对照 PRD 检查：
  1. 是否有遗漏的验收标准
  2. 实现步骤是否有逻辑冲突
  3. 是否违反项目架构约定（见 Architecture.md）
  [粘贴 plan.md 内容]
```

**将 Review 反馈带回 Claude Code**：
```
你（对 Claude Code）：
  根据以下 Review 反馈修改计划：
  [粘贴 Codex 的反馈]
```

**TDD 建议**：在计划阶段就让 Agent 写出测试用例。测试即验收标准的代码化表达，Agent 实现时可以自动验证。

**产出物**：`DesignDoc/PLAN_XX_模块名.md`

---

### 阶段四：实现

**目标**：Agent 按照 Plan 编写代码。

**操作方式**：

**单任务实现**：
```
你（对 Claude Code）：
  请按照 DesignDoc/PLAN_01_MAP_UI.md 的步骤实现。
  每完成一个步骤，运行测试确认通过后再继续下一步。
```

**并行实现（Worktree 方式，适用于无依赖的多个子任务）**：
```bash
# 终端 1：实现子任务 A
git worktree add ../project-task-a feature/task-a
cd ../project-task-a && claude
> 请按照 PLAN 的步骤 1-3 实现 MapPanel...

# 终端 2：实现子任务 B（与 A 无依赖）
git worktree add ../project-task-b feature/task-b
cd ../project-task-b && claude
> 请按照 PLAN 的步骤 4-6 实现 MapNodeUI...
```

**并行开发的前提**：
- 子任务之间无文件交叉修改
- 每个子任务有独立的验收标准
- 合并时通过 `git merge` 整合

**开发者在这个阶段做什么**：
- 不需要盯着 Agent 写代码
- 可以同时启动多个 Agent 处理不同子任务
- 在一个 Agent 运行时，Review 另一个 Agent 的产出
- 关注 Agent 卡住或反复报错的情况，及时介入

---

### 阶段五：功能测试 + 迭代

**目标**：验证实现是否符合 PRD，修正偏差。

**步骤**：
1. **Agent 自查**：让 Agent Review 自己的代码，对照验收标准检查
   ```
   你：请对照 PRD_01_MAP_UI.md 的验收标准，Review 你刚写的代码，列出可能不符合的地方
   ```

2. **人工测试**：在 Unity 中运行游戏，按交互方式逐步操作
   - 对照 PRD 的验收标准逐项勾选
   - 记录不符合的地方

3. **迭代修正**：
   - 如果是代码 Bug → 直接让 Agent 修复
   - 如果是需求偏差（PRD 没说清楚）→ **先修改 PRD，再让 Agent 按更新后的 PRD 修改代码**
   - 不要口头描述修改意图然后让 Agent 改——这会导致 PRD 和代码不一致

**关键原则**：发现偏差时，从文档开始修改，而不是直接改代码。文档是 single source of truth。

---

### 阶段六：人工 Review

**目标**：确保代码质量、架构一致性，防止 Agent 引入隐患。

**步骤**：

1. **让 Agent 划出 Review 重点**：
   ```
   你：请列出这次改动中最需要人工仔细 Review 的部分，包括：
   - 与其他系统的交互点
   - 状态管理和生命周期
   - 可能的空引用或竞态
   - 偏离架构约定的地方
   ```

2. **基于清单 Review**：聚焦 Agent 列出的重点，不逐行读所有代码

3. **Review 关注点**：

   | 关注 | 不关注 |
   |------|-------|
   | 架构是否符合 Architecture.md | 变量命名风格 |
   | 系统交互点是否正确 | 代码格式 |
   | 是否有 FindObjectOfType 反模式 | 注释是否充分 |
   | 事件订阅是否有对应的取消订阅 | 具体算法实现 |
   | 空引用和边界情况 | 性能微优化 |

4. **反馈给 Agent**：
   ```
   你：以下是 Review 发现的问题，请修改：
   1. MapPanel.cs 第 45 行用了 FindObjectOfType，应通过 Inspector 引用
   2. 节点点击事件没有取消订阅，OnDestroy 中需要清理
   ```

---

### 收尾：让 Agent 补测试

功能完成并 Review 通过后，让 Agent 为核心逻辑补充单元测试：

```
你：请为 MapPanel.cs 和 EncounterManager.cs 编写单元测试，覆盖 PRD 中的验收标准。
    测试应独立于 Unity 场景运行（纯逻辑测试，不依赖 MonoBehaviour）。
```

Agent 写的测试不完美，但**有和没有的差距很大**。这是零成本高回报的操作。

## 4. 工作流二：原型主导

适用于小功能或需求不明确的情况。

**流程**：
```
模糊想法 → 直接对话让 Agent 实现 → 手动测试 → 迭代对话修改
    │
    └─ 复杂度膨胀？
         ├─ 否 → 完成，直接合并
         └─ 是 → Agent 阅读已有代码，生成 PRD → 切换到工作流一
```

**操作方式**：
```
你：帮我在 BattleManager 里加一个逃跑按钮，点击后结束战斗回到地图
Agent：[直接写代码]
你：可以用了，但逃跑后应该扣 10% 最大 HP
Agent：[修改代码]
你：好的，合并
```

**切换信号**——以下情况说明复杂度已膨胀，应切换到文档主导：
- 对话超过 10 轮仍在改需求
- 涉及 3 个以上文件的交叉修改
- 你发现自己说不清"到底要什么效果"

**切换方法**：
```
你：请阅读你刚才写的所有代码，生成一份 PRD，包含术语表、交互方式和验收标准，
    输出到 DesignDoc/PRD_XX_功能名.md
```

## 5. 文档生命周期

不是所有文档都需要长期维护。用二维坐标判断：

```
                    容易描述          难以描述
              ┌──────────────┬──────────────┐
  核心系统     │  Q1: 持续维护  │  Q2: 视情况   │
              │  （卡牌系统、   │  （复杂AI行为、│
              │   战斗系统）    │   程序化生成） │
              ├──────────────┼──────────────┤
  非核心系统   │  Q4: 跳过文档  │  Q3: 用完即弃 │
              │  （UI微调、     │  （一次性脚本、│
              │   配置修改）    │   数据迁移）  │
              └──────────────┴──────────────┘
```

| 象限 | 策略 | 示例 |
|------|------|------|
| Q1 | PRD 随代码同步更新，每次修改先更新 PRD | 战斗系统、地图系统、卡牌架构 |
| Q2 | 写 PRD 辅助开发，完成后视复杂度决定是否维护 | 敌人 AI 状态机、事件效果 |
| Q3 | 写 PRD 辅助开发，完成后归档不再维护 | 数据导入脚本、一次性重构 |
| Q4 | 不写 PRD，直接用原型主导工作流 | 调整按钮位置、改颜色 |

**本项目的核心系统（Q1，需持续维护文档）**：
- `Architecture.md` — 整体架构
- `PRD_00_MASTER.md` — 项目总体需求
- 各核心模块 PRD（地图、战斗、卡牌）

## 6. 分支与并行开发

### 6.1 分支策略

```
main (稳定版本)
  │
  ├── feature/map-ui          ← PRD-01 的实现分支
  ├── feature/encounter       ← PRD-02 的实现分支
  ├── feature/endgame-panel   ← PRD-03 的实现分支
  └── feature/event-panel-ui  ← PRD-04 的实现分支
```

**规则**：
- 每个 PRD 对应一个 feature 分支
- 分支从 main 创建，完成后合并回 main
- 合并前必须通过阶段六（人工 Review）

### 6.2 Worktree 并行开发

当多个 PRD 之间无文件交叉时，可用 Git Worktree 并行开发：

```bash
# 主仓库保持在 main
# 为每个任务创建独立工作目录

git worktree add ../project-map-ui feature/map-ui
git worktree add ../project-encounter feature/encounter
git worktree add ../project-endgame feature/endgame-panel

# 每个目录启动一个 Claude Code 实例
# 终端 1
cd ../project-map-ui && claude

# 终端 2
cd ../project-encounter && claude

# 终端 3
cd ../project-endgame && claude
```

**优势**：
- 一个 Agent 在跑时，你可以 Review 另一个 Agent 的产出
- 降低等待空转时间
- 每个 Agent 有独立的文件系统，不会互相冲突

**合并**：
```bash
git checkout main
git merge feature/map-ui
git merge feature/encounter        # 解决冲突（如有）
git merge feature/endgame-panel

# 清理
git worktree remove ../project-map-ui
git worktree remove ../project-encounter
git worktree remove ../project-endgame
```

### 6.3 什么时候不要并行

- 两个任务修改同一个文件（如都改 BattleManager.cs）→ 串行开发
- 任务 B 依赖任务 A 的产出 → 串行开发
- 你无法同时 Review 多个 Agent 的产出 → 减少并行数

## 7. 工具使用规范

### 7.1 主力工具

| 工具 | 用途 |
|------|------|
| **Claude Code** | 计划制定、代码实现、代码 Review、写测试 |
| **Codex / ChatGPT** | Review Claude Code 的计划（交叉验证） |

保持工具链轻量，不在工具选择上过度投入。

### 7.2 固定流程 Skill 化

将重复性操作脚本化，让 Agent 而非你来执行：

```
# 示例：让 Agent 创建编译+测试的一键脚本
你：请写一个 bash 脚本 scripts/build_and_test.sh，功能：
    1. 用 Unity batch mode 编译项目
    2. 运行所有 EditMode 测试
    3. 输出测试结果摘要
```

常见可 Skill 化的流程：
- 编译检查
- 运行测试
- 代码格式化
- 生成模块进度报告

### 7.3 Agent 使用技巧

- **给 Agent 完整上下文**：让它先读 Architecture.md 和相关 PRD，再开始工作
- **分步而非一次性**：不要一次给 Agent 整个 PRD 让它全部实现，按步骤走
- **让 Agent 自查**：实现完成后让 Agent 自己对照验收标准 Review
- **不过度关注代码细节**：关注架构和系统交互点，具体写法交给 Agent

## 8. 团队协作约定

### 8.1 模块拆分原则

- 每个模块有清晰的边界和独立的 PRD
- 模块之间通过**接口**（事件、方法签名）通信，不直接持有内部引用
- 共享状态尽量少，集中在 `RunState` 和 `BattleContext`

### 8.2 团队协调重心

```
传统团队：一起写代码 → 解决冲突 → Code Review
AI 团队：定义模块接口 → 各自用 Agent 实现 → 在接口层 Review
```

协调重心从"一起写代码"转为"定义模块接口"。

### 8.3 沟通规范

- 修改了 PRD → 通知相关模块的负责人
- 修改了模块接口 → 更新 Architecture.md 并通知所有人
- 发现 Agent 产出的模式性问题 → 记录到 CLAUDE.md 作为全局指令

### 8.4 Review 分工

| Review 类型 | 谁来做 | 关注什么 |
|------------|--------|---------|
| Plan Review | 交叉 Review（另一个 Agent 或另一个人） | 逻辑完整性、与 PRD 一致性 |
| Code Review | 模块负责人 | 架构合规、接口正确性 |
| 功能测试 | 任意团队成员 | 交互体验、验收标准 |
