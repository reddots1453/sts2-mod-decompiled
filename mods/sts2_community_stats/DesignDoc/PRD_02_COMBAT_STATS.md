# PRD-02: 战斗统计修复与测试

> 版本：1.0 | 日期：2026-04-07  
> 依赖：CombatTracker、ContributionMap、CombatHistoryPatch（均已实现）

---

## 1. 概述

战斗贡献统计是本 Mod 最复杂的系统（1900+ 行补丁代码、60+ 力量钩子、35 遗物钩子）。经人工测试发现多处数据遗漏和不准确。本 PRD 定义系统性测试矩阵和修复标准，确保所有已知场景下统计数据准确。

**原则**：不新增统计维度，只修复和补全现有统计项的准确性。

---

## 2. 术语表

| 术语 | 定义 |
|------|------|
| **贡献类别** | 伤害（Damage）、防御（Defense）、抽牌（Draw）、能量（Energy）、星星（Stars）、治疗（Healing）六大类 |
| **贡献来源类型** | 卡牌（Card）、遗物（Relic）、药水（Potion）三种 |
| **FIFO 格挡池** | 格挡归因的先进先出队列：先获得的格挡先被消耗，消耗时归因到对应来源 |
| **LIFO Osty 栈** | Osty HP 归因的后进先出栈：最后回复的 HP 先被伤害消耗 |
| **上下文栈** | ContributionMap 中追踪当前"谁在生效"的栈结构，用 Prefix 入栈、Postfix 出栈 |
| **末杀修正** | 最后一击伤害的特殊处理：敌人死亡时 `IsEnding` 标记会导致常规补丁跳过，需 KillingBlowPatcher 补偿 |
| **测试矩阵** | 按角色 × 来源类型 × 贡献类别的系统性测试覆盖表 |
| **手动验证** | 在游戏中实际操作并对照贡献面板数据的测试方法 |

## 3. 交互方式

### 3.1 测试执行方式

由于 Harmony 补丁依赖游戏运行时，无法做传统单元测试。测试通过以下方式进行：

```
设计测试场景（使用 DevTools Mod 构造牌组/遗物/敌人）
  → 进入战斗，执行预定操作
  → 战斗结束后查看贡献面板
  → 手动比对预期值与实际值
  → 不匹配的场景记录为 Bug → 修复 → 回归测试
```

### 3.2 测试场景构造

使用 `sts2_dev_tools` Mod（F7 解锁全部）+ 游戏内自定义模式构造：
- 选择特定角色
- 通过自定义牌组控制手牌
- 选择特定遭遇控制敌人
- 装备特定遗物/药水

### 3.3 验证数据读取

贡献面板在战斗结束后自动弹出，显示：
- **本场战斗** Tab：本次战斗的贡献明细
- **本局汇总** Tab：累计所有战斗的贡献

验证时需检查：
- 每个来源的贡献数值是否正确
- 子条目（Sub-bar）是否正确归因
- 各类别总和是否与战斗日志一致
- 遗漏的来源是否出现在列表中

## 4. 测试矩阵

### 4.1 基础伤害归因

| # | 场景 | 操作 | 预期 |
|---|------|------|------|
| D1 | 单体攻击卡 | 打出"打击"对敌人造成伤害 | 打击的 DirectDamage = 实际伤害值 |
| D2 | AOE 攻击卡 | 打出旋风斩对所有敌人造成伤害 | 旋风斩的 DirectDamage = 所有敌人受到伤害之和 |
| D3 | 多段攻击卡 | 打出"连击"命中 3 次 | 连击的 DirectDamage = 3 次伤害之和 |
| D4 | 末杀伤害 | 用一张卡打出 20 伤害，敌人只剩 5 HP | 该卡的 DirectDamage = 5（实际造成），而非 20（面板值） |
| D5 | 力量加成 | 有 3 力量时打出"打击"（基础 6）| 打击 DirectDamage = 6, ModifierDamage = 3（力量贡献）|
| D6 | 易伤加成 | 敌人有易伤时打出"打击"| 施加易伤的来源获得 ModifierDamage（50% 加成部分）|
| D7 | 中毒伤害 | 施加 10 中毒，敌人回合受到毒伤 | 施加中毒的来源获得 AttributedDamage = 毒伤值 |
| D8 | 荆棘伤害 | 有荆棘效果，敌人攻击触发反伤 | 荆棘来源获得 AttributedDamage |
| D9 | 升级增量 | 打击升级后（6→9），打出一次 | 打击 DirectDamage = 6, UpgradeDamage = 3 |
| D10 | 遗物被动伤害 | 水银沙漏每回合对所有敌人造成 3 伤害 | 水银沙漏 DirectDamage += 3 × 敌人数 |

### 4.2 防御归因

| # | 场景 | 操作 | 预期 |
|---|------|------|------|
| B1 | 基础格挡 | 打出"防御"获得 5 格挡，敌人攻击 3 | 防御的 EffectiveBlock = 3（被消耗的部分）|
| B2 | 格挡溢出 | 打出"防御"获得 5 格挡，敌人攻击 10 | 防御的 EffectiveBlock = 5（全部消耗）|
| B3 | FIFO 顺序 | 先打"防御A"5 格挡，再打"防御B"5 格挡，敌人攻击 7 | A 的 EffectiveBlock = 5, B 的 EffectiveBlock = 2 |
| B4 | 灵巧加成 | 有 2 灵巧时打出"防御"（基础 5）| 防御 EffectiveBlock 基于 7 格挡，灵巧来源获得 ModifierBlock |
| B5 | 虚弱减伤 | 敌人有虚弱（-25% 攻击），攻击 20 | 施加虚弱的来源获得 MitigatedByDebuff = 5 |
| B6 | Buffer 减伤 | 有 Buffer 1 层，敌人攻击 20 | Buffer 来源获得 MitigatedByBuff = 20 |
| B7 | 无实体减伤 | 有无实体，敌人攻击 20 | 无实体来源获得 MitigatedByBuff = 19（伤害降为 1）|
| B8 | 力量削减 | 降低敌人 2 力量后，敌人攻击 | 削减来源获得 MitigatedByStrReduction = 避免的伤害 |
| B9 | 遗物被动格挡 | 蛇眼（Orichalcum）回合结束时获得 6 格挡（如果无格挡）| 蛇眼 EffectiveBlock = 消耗量 |
| B10 | 自伤 | 打出放血（自伤 3 HP）| 放血 SelfDamage = 3，显示为红色负防御 |

### 4.3 Osty 特殊场景（STS2 特有）

| # | 场景 | 操作 | 预期 |
|---|------|------|------|
| O1 | Osty 吸收伤害 | Osty 替玩家承受伤害 | 伤害消耗 Osty HP 的 LIFO 栈，归因到回复 Osty 的来源 |

### 4.4 特殊卡牌/力量交互

| # | 场景 | 操作 | 预期 |
|---|------|------|------|
| S1 | 追索之刃（SeekingEdge）| 对非主目标的伤害 | 归因到 SeekingEdge 而非 SovereignBlade |

## 5. 验收标准

### AC-01: 伤害归因准确
- [ ] D1-D10 全部场景通过手动验证
- [ ] 总伤害 = DirectDamage + AttributedDamage + ModifierDamage + UpgradeDamage
- [ ] 无来源被遗漏（战斗中造成的每一点伤害都出现在图表中）
- [ ] 末杀伤害不被重复计算

### AC-02: 防御归因准确
- [ ] B1-B10 全部场景通过手动验证
- [ ] FIFO 格挡池消耗顺序正确
- [ ] 自伤显示为红色负值
- [ ] 虚弱/Buffer/无实体减伤归因到正确来源

### AC-03: 特殊机制准确
- [ ] O1-O3（Osty）场景通过验证
- [ ] S1-S4（特殊交互）场景通过验证
- [ ] 子条目（Sub-bar）正确显示来源关系

### AC-04: 跨角色基础覆盖
- [ ] 6 个角色各至少完成一局完整战斗验证
- [ ] 每个角色的特色机制（中毒/充能球/姿态/星星/召唤物）归因正确

### AC-05: 数据一致性
- [ ] "本场战斗"Tab 数据与战斗中实际发生一致
- [ ] "本局汇总"Tab 数据 = 所有"本场战斗"数据之和
- [ ] 上传到服务器的 ContributionUpload 数据与本地面板显示一致

### AC-06: 不崩溃
- [ ] 所有测试场景中 Mod 不导致游戏崩溃
- [ ] 遇到未处理的力量/遗物时，graceful 降级（记录日志，不崩溃），贡献显示为"未追踪"

## 7. 技术约束

- 测试只能通过游戏内手动操作进行（Harmony 补丁不支持脱离游戏的单元测试）
- 可借助 DevTools Mod 构造测试场景（解锁全部卡牌/遗物）
- 修改仅限 `src/Collection/` 和 `src/Patches/` 目录
- 不新增贡献类别，不修改 ContributionAccum 的字段结构
- 每次修复后需回归验证之前通过的场景
