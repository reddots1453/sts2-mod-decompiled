# PRD-01: 服务器部署与端到端验证

> 版本：1.0 | 日期：2026-04-07  
> 依赖：后端代码（已完成）、Docker Compose 配置（已完成）、Nginx/SSL 配置（已完成）

---

## 1. 概述

将已完成的后端代码部署到云服务器，验证从客户端上传数据到查询统计的完整数据流，确保服务器在真实环境下稳定运行。

---

## 2. 术语表

| 术语 | 定义 |
|------|------|
| **VPS** | 云端虚拟私人服务器，运行后端服务 |
| **Docker Compose** | 容器编排工具，一键启动 API + PostgreSQL + Redis + Nginx + Certbot 五个服务 |
| **SSL/TLS 证书** | HTTPS 所需的加密证书，通过 Let's Encrypt + Certbot 自动签发 |
| **预计算任务** | APScheduler 后台定时任务，每 10 分钟聚合统计数据写入 Redis |
| **健康检查端点** | `GET /health`，返回 API、数据库、Redis 的可用状态 |
| **种子数据** | 通过 `seed_test_data.py` 生成的模拟局数据，用于验证聚合逻辑 |
| **离线队列** | 客户端在服务器不可达时缓存上传请求，服务恢复后自动重传 |
| **限流** | 对上传（10次/分钟）和查询（60次/分钟）的请求频率限制 |

## 3. 交互方式

### 3.1 部署流程

```
准备 VPS（Linux） 
  → 安装 Docker + Docker Compose
  → 克隆仓库 / 上传 server/ 目录
  → 配置 .env（数据库密码、域名）
  → 配置域名 DNS A 记录指向 VPS IP
  → docker compose up -d
  → Certbot 自动签发 SSL 证书
  → 验证 https://域名/health 返回 healthy
```

### 3.2 部署后验证——端到端数据流

#### 步骤 1：健康检查
```
GET https://域名/health
期望响应：{"status": "healthy", "db": "ok", "redis": "ok"}
```

#### 步骤 2：种子数据上传
```bash
cd server/
python scripts/seed_test_data.py --runs 100 --api-url https://域名
```
- 验证 100 条记录成功写入
- 验证数据库 `runs` 表有 100 条记录

#### 步骤 3：统计查询
```
GET https://域名/v1/stats/bulk?char=IRONCLAD&min_asc=0&max_asc=20
期望：返回 BulkStatsBundle，cards/relics/events/encounters 非空
```

#### 步骤 4：客户端真实上传
1. 修改 `config.json` 的 `api_base_url` 为 `https://域名/v1`
2. 在 STS2 中完成一局游戏
3. 检查服务器日志确认收到上传请求
4. 检查数据库确认局数据完整写入

#### 步骤 5：客户端查询验证
1. 开始新一局，进入卡牌奖励界面
2. 确认卡牌下方显示选取率/胜率标签
3. 悬停遗物确认显示社区胜率
4. 进入事件确认显示选项统计

#### 步骤 6：异常场景验证
1. 关闭服务器 → 客户端完成一局 → 确认数据进入离线队列 → 重启服务器 → 确认队列数据重传成功
2. 快速连续上传 15 次 → 确认限流生效（返回 429）
3. 发送畸形数据 → 确认服务器返回 422 且不崩溃

### 3.3 客户端配置切换

Mod 需要将 `config.json` 从开发环境切换到生产环境：

```json
// 开发环境
{ "api_base_url": "http://localhost:8080/v1" }

// 生产环境
{ "api_base_url": "https://实际域名/v1" }
```

发布时 `config.json` 应默认指向生产环境。

## 4. 验收标准

### AC-01: 服务可用
- [ ] `GET /health` 返回 `{"status": "healthy", "db": "ok", "redis": "ok"}`
- [ ] HTTPS 证书有效（浏览器无安全警告）
- [ ] 服务在 `docker compose up -d` 后自动启动，宿主机重启后自动恢复

### AC-02: 数据上传
- [ ] 种子脚本 100 条数据全部成功写入
- [ ] 客户端真实游戏数据成功上传
- [ ] 上传请求响应时间 < 2 秒（正常网络条件下）
- [ ] 畸形数据被拒绝（422），服务器不崩溃

### AC-03: 统计查询
- [ ] `/v1/stats/bulk` 返回完整的 BulkStatsBundle
- [ ] 预计算任务每 10 分钟执行一次，Redis 中有缓存数据
- [ ] 查询响应时间 < 500ms（缓存命中时）
- [ ] 6 个角色 × 5 个升级范围均有预计算数据

### AC-04: 客户端集成
- [ ] 卡牌奖励界面显示选取率/胜率
- [ ] 遗物悬停显示社区胜率
- [ ] 事件选项显示社区选择率
- [ ] 筛选面板（F9）修改后统计标签刷新

### AC-05: 容错与安全
- [ ] 服务器不可达时，客户端不崩溃，数据进入离线队列
- [ ] 离线队列数据在服务恢复后自动重传
- [ ] 限流生效：上传 > 10次/分钟 返回 429
- [ ] PostgreSQL 和 Redis 端口不对外暴露
- [ ] 无敏感信息泄露（无 Steam ID、系统信息等）

### AC-06: 运维基础
- [ ] 日志输出到标准输出，可通过 `docker compose logs` 查看
- [ ] 数据库有持久化卷挂载，容器重建数据不丢失
- [ ] Redis 有 RDB 持久化

---

## 5. 技术约束

- VPS 最低配置：2 核 CPU、4GB 内存、40GB SSD
- 操作系统：Ubuntu 22.04+ 或 Debian 12+
- 域名需提前购买并配置 DNS
- 已有部署指南：`DEPLOYMENT_GUIDE.md`（32.6KB）
- 客户端修改仅 `config.json` 一个文件
- 不修改任何后端代码（代码已完成），仅做部署和验证
